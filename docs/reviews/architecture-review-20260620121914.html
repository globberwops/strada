Problem: Geometry evaluation and coordinate projection are shallow, forcing the compiled physics model to manually dispatch across analytical helpers and numerical solvers.

Solution: Consolidate geometry math and projection searches behind a deep ReferenceLine module with a clean evaluation and projection interface.

Wins

  • Locality: geometry mathematics concentrate in one module.
  • Leverage: index query uses one unified interface.
  • Tests verify geometry through a direct seam.
  • Simplifies the spatial index querying code.

Problem: Rotation math functions are shallow utility wrappers exposing raw matrix data layouts and forcing callers to manually orchestrate compositions and transpositions.

Solution: Replace raw matrix manipulation with a deep Rotation module hiding algebraic details and exposing high-level composition and coordinate transformation methods.

Wins

  • Depth: raw matrix representation is hidden inside the module.
  • Leverage: coordinate frame transformations write like text.
  • Locality: gimbal lock limits are resolved in one place.

Top Recommendation

Our top recommendation is to Consolidate Reference Line Evaluation and Projection (Candidate 1). By taking the reference line compilation data structure and pairing it directly with its evaluation algorithms and projection solver, we transition the geometry code from a series of shallow helper functions into a deep, cohesive module. This increases mathematical locality, eliminates inline projection math inside the spatial search, and provides a clear, mockable test surface for the core geometric projection engine.
