Problem

Both ElevationProfile and LaneNetwork duplicate the implementation of PolynomialsSoA compilation and evaluation, scattering optimizations and violating locality.

Solution

Extract vertical and lateral profile polynomial math into a single dedicated, deep Polynomials module.

Wins

  • locality: optimization rules live in one module
  • leverage: one optimized implementation serves multiple clients
  • interface shrinks: internal duplicates are removed
  • locality: tests exercise a single mathematical interface

Problem

Tessellator duplicates AST-to-compiled lane handle mapping logic, bypassing the CPM seam and causing architectural friction.

Solution

Expose a pre-compiled lane lookup method FindLaneId on the CompiledPhysicsModel interface.

Wins

  • locality: indexing rules reside in compiled map
  • leverage: callers resolve IDs with one method
  • locality: AST inspection logic is eliminated
  • interface: cleaner boundary between AST and CPM

Problem

CompiledPhysicsModel::Build performs raw AST lane geometry sampling loops, making the orchestration function shallow and highly coupled to raw AST schemas.

Solution

Move road-width estimation into LaneNetwork::Build and feed pre-computed widths directly into ReferenceLine AABB creation.

Wins

  • locality: width logic stays in lane network
  • interface: CPM orchestration is simple
  • locality: tests verify width compiler in isolation
  • leverage: ReferenceLine builds AABBs without AST

Top Recommendation

Introduce a Lane Resolution Seam on the Compiled Map

We recommend implementing Candidate 2 (Introduce a Lane Resolution Seam on the Compiled Map) first. Currently, the Tessellator is forced to replicate the internal indexing order rules of the Compiled Physics Model by performing manual AST traversal loops. Resolving this via a unified FindLaneId seam on CPM/LaneNetwork eliminates this leak, aligns the codebase with our design guidelines, and improves test isolation.
